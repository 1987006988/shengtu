# AGENTS

本文档定义 `升途` / `Ascendant Path` 的长期协作规范。所有参与本仓库维护的人类或代理，都应遵守以下规则。

## 项目命名

- 项目中文总名：`升途`
- 英文正式名：`Ascendant Path`
- 仓库 / 目录英文名：`shengtu`
- 第一部标题：`升途：封绝之地`
- 若本地物理路径仍显示旧工程名，文档与协作口径仍统一使用 `shengtu`；物理目录重命名需由人工在文件系统层执行。

## 核心原则

- 术语统一：核心术语、模块术语、别名与暂定用法，统一记录在 `worldbible/09_reference/glossary.md`。
- 禁止擅自新增核心设定：任何新的底层机制、世界级规则、核心世界结构、关键历史锚点或主干阵营框架，未经确认不得写入正典文件。
- 重大变更必须记录：凡是会影响多个模块、术语含义、世界底层逻辑、剧情前提或角色定位的变更，必须记录到 `worldbible/09_reference/decision_log.md`。
- 优先小步迭代：每次只推进一个模块或一个问题，先拍板再扩写，避免并行发散。
- 保持仓库总范围：不得把整个仓库逐步写窄成只服务 `古裂残天` 的单世界仓库。
- 保持第一部尺度：`升途：封绝之地` 的当前舞台是 `古裂残天` 内部隔绝地域，不得默认写成整个主世界命运线。

## 范围判断规则

在扩写任何内容前，必须先判断该内容属于哪一层：

1. 宇宙通则层
2. 九主世界 / 主世界层
3. 故事主世界层
4. 游戏聚焦层

并遵守以下约束：

- 若内容属于宇宙通则层，不得只按 `古裂残天` 局部经验回写到全宇宙文件。
- 若内容属于九主世界 / 主世界层，应先写主世界总层与比较框架，再落到故事主世界层。
- 若内容仅属于游戏聚焦层，不得写入宇宙通则文件。
- 若内容仅服务主角团、具体角色或具体主线，不得反向定义整个宇宙通则。

## 写作边界

- 当前阶段只允许创建、整理或修订结构、模板、索引、规范与正式设定文档。
- 不要发明未经确认的势力名、地名、人物名、小世界名、历史事件名与具体职业细节。
- 不要把讨论中的提案直接写成正典。
- 不要让单个模块越权决定其他模块的底层规则。
- 不要让 `04_main_world` 直接退化为 `古裂残天` 专属目录。
- 不要让 `07_characters` 与 `08_story` 的游戏聚焦内容反向吞没整个仓库。
- 当前下一步是“白鹿原势力与关键节点落位阶段”，不是任务层。
- 进入任务层前，必须先回查 `worldbible/00_project_overview/06_stage_gates.md` 与 `worldbible/00_project_overview/08_pre_task_layer_requirements.md`，确认前置条件已满足。

## 文件优先级

当不同文件出现冲突时，按以下顺序处理：

1. `worldbible/09_reference/canon_rules.md`
2. `worldbible/09_reference/glossary.md`
3. `worldbible/09_reference/decision_log.md`
4. 对应模块的正式文档
5. 草稿、研究笔记与待定记录

## 变更流程

1. 先确认本轮只处理一个明确模块或一个明确问题。
2. 先检查该模块 `00_index.md` 的范围、状态与交叉引用。
3. 再检查 `worldbible/00_project_overview/05_scope_map.md`，确认内容所属层级。
4. 优先在正式文档中落实已经拍板的结论，不依赖会话记忆。
5. 检查本次变更是否影响术语表、正典规则、决策日志或状态页。
6. 若影响跨模块口径，必须同步更新参考文件并登记决策日志。
7. 最后补充索引、当前状态与交接摘要。

## 文档要求

- 文件命名应稳定、清晰、可扩展。
- 索引文件负责汇总与导航，不承载过多正文细节。
- 所有待定内容必须显式标记为“待定”或“待人工拍板”。
- 重大结论尽量附带影响范围、关联术语与确认状态。
- 范围说明应优先于正文扩写，以避免后续跑偏。

## 协作要求

- 以仓库沉淀为准，不依赖外部聊天记忆。
- 发现冲突时，先暂停扩写，优先修正术语、边界和参考文件。
- 对跨模块内容，优先建立引用关系，而不是复制粘贴。
- 若当前重点对象是 `古裂残天`，仍需持续提醒：它只是当前重点展开对象，不是整个仓库的唯一对象。
