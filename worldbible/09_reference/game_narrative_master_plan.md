# 《升途：封绝之地》大型 CRPG 叙事文本总工程计划

## 0. 文件定位

本文档是《升途：封绝之地》大型 CRPG 叙事文本总工程的项目级总蓝图。

它不是一次性执行 prompt，不用于在单轮会话中直接产出主线任务文本、同伴文本、支线文本或系统反馈文本。它是后续批次化执行的母本，用于定义文本资产规模、阶段顺序、拆包规则、验收标准、长期记忆同步方式与边界护栏。

本文档来源于外部输入《升途：封绝之地》大型 CRPG 叙事文本总工程计划，经最小必要整理后正式落库。后续若同名文件已存在，应按更新逻辑处理：先核对外部输入与仓库当前状态，再以本文件为母本继续修订，而不是把旧会话记忆当作事实来源。

## 1. Summary

本轮不再把 `白鹿原` 叙事承载误收口为“第一卷小说完成稿”，而是把现有故事成果降级为“上游基线素材”，继续扩建为可长期支撑大型强叙事 RPG 的文本资产工程。

执行目标是：以 `白鹿原` 为稳定尺度，在不改写已锁定正典与护栏的前提下，形成主线、支线、同伴、区域、Codex、Journal、物品、系统反馈、分支状态与失败反馈共同运转的多层文本库，并把当前项目状态从“故事完成稿”切换为“第一款游戏叙事文本总工程”。

已锁定的实施决策：

- 你指定的主文件全部保留为总入口或主控文件。
- 允许在同目录新增拆包子文件承载正文资产，主文件负责总览、索引、规则、映射和跨包引用。
- 既有 `30_book1_story_draft.md` 与 `31_book1_story_revised.md` 不删除，改作“小说化参考底稿”，不再冒充游戏文本终稿。

## 2. Key Changes

### 2.1 总控层重建

先新增两个项目级总控文件：

- `worldbible/09_reference/game_narrative_master_plan.md`
- `worldbible/09_reference/game_narrative_master_plan_review.md`

这两个文件必须明确：

- 为什么必须从“小说短稿”切换为“CRPG 叙事文本总工程”。
- 文本总量目标：最低 `1,500,000` 汉字，目标区间 `3,000,000–6,000,000` 汉字。
- 资产分类占比和阶段推进规则。
- 白鹿原范围边界：只做第一款游戏的白鹿原阶段，不上浮为整个 `古裂残天` 大全。
- 为什么“卷一小说”不能替代任务文本、分支反馈、Journal / Codex、同伴包和环境文本。

### 2.2 文件组织规则

采用“锚点文件 + 拆包子文件 + 总索引”的稳定结构。

锚点文件：

- 使用指定主文件作为阶段总控、矩阵、规则、索引、汇总入口。
- 每个大体量文本包允许继续拆为同目录子文件，命名保持稳定前缀。

默认拆包规范：

- 主线文本包：`33_main_quest_text_pack_part1.md` 等为入口，再按 `act` 或任务组拆成 `33a_... / 33b_...`。
- 同伴文本包：`18/19/20` 为入口，再按角色或场景群拆包。
- 支线与区域文本包：`37/38/39` 为入口，再按区域带和功能群拆包。
- Codex / Journal / Items / System：`40/41/42/43` 为入口，再按分类拆包。
- 如新增拆包，必须在对应锚点文件顶部维护索引和字数统计表。

统一文档接口：

- 每个任务单元必须有：`ID / 前置 / 目标 / 关键冲突 / 关键选择 / 输出状态 / 关联区域 / 关联势力 / 关联同伴 / Journal 更新`。
- 每张场景卡必须有：`存在理由 / 推动冲突 / 增权对象 / 前置状态 / 变体结果 / 输出状态`。
- 每条文本资产必须标注归属层：`主线 / 支线 / 同伴 / 区域 / Codex / Journal / Item / System`。
- 状态机文件统一使用“状态触发条件 + 可见反馈 + 对其他系统影响”的写法，不写死板流程图。

## 3. 七阶段实施顺序

### 3.1 阶段 0：总控总计划

产出：

- `game_narrative_master_plan.md`
- `game_narrative_master_plan_review.md`

核心动作：

- 把现有“故事完成稿”定位为上游素材，不再作为终点。
- 立文本资产分类表、字数预算表、拆包规则、阶段门槛和最终验收表。
- 在计划里写明每阶段必须同步 `decision_log.md`、`04_current_state.md`、`session_handoff.md`。

切换门槛：

- 已明确不是“再写一部长篇小说”，而是游戏文本工程。
- 已明确拆包规则和总量达成路径。
- 已明确白鹿原尺度边界和白鹿之灵边界。

### 3.2 阶段 1：重建游戏级主线架构

重做并扩充：

- `25_volume1_core_conflict.md`
- `26_volume1_revelation_ladder.md`
- `27_volume1_plot_spine.md`
- `28_volume1_chapter_outline.md`
- `29_volume1_scene_cards.md`
- 新增 `game_phase1_review.md`

实施标准：

- 从“16 章短卷”改造成 `Act 1 / Act 2 / Act 3 / Epilogue`。
- 主线章节 / 任务节点单元扩到 `40–80`。
- 场景卡扩到 `120–240`。
- 每个 Act 都要写清主目标、次目标、冲突升级、揭示层级、同伴推进、区域推进、阵营态度变化。
- 场景卡必须含前置状态与变体结果，能直接支撑任务设计和演出设计。

最小修补重点：

- 防止主线仍按小说脊柱压缩。
- 防止只写推进摘要、不写状态入口。

### 3.3 阶段 2：势力状态机与区域状态机

- 显式写清哪些变化由剧情强推，哪些可由玩家选择改写。

最小修补重点：

## 5. Test Plan

必须在每阶段 review 文件中执行这些检查：

- 范围检查：是否仍限定在 `白鹿原` 第一款游戏尺度，没有把第一部无声上浮为整个 `古裂残天`。
- 正典检查：是否改写了未获确认的宇宙通则、主世界长期正典或新核心专名。
- 白鹿之灵检查：是否出现万能解释、万能支援、万能真相机、无代价复生。
- 状态机检查：主线、支线、区域、同伴、系统反馈是否能被相互调用。
- 文本厚度检查：不是示例，不是摘要，不是小说附录；每层资产都达到最低量级。
- 任务可实现性检查：任务单元、场景卡、Journal、反馈文本能否直接供叙事设计与任务设计引用。
- 质量检查：信息获取、行动成本、失败结果、代价承担、阵营行为、角色选择都符合内部现实逻辑。
- 分支检查：关键节点必须存在状态差异、态度差异、文本差异，不允许只有表面选项。
- 总量检查：按阶段累计统计字数，确保最终达到至少 `1,500,000` 汉字。

最终验收必须给出：

- 总字数与各层占比。
- 哪些层已足够支撑前期开发。
- 哪些层仍可继续扩容但不妨碍进入下一阶段。
- 是否建议项目正式转入“游戏叙事生产期”。

## 6. Assumptions

- 现有 `story_master_execution_plan.md` 和 `30/31` 故事稿继续保留，但本轮会明确把它们降为上游参考，不再作为项目终点。

- 默认不引入大量新正式专名；如某些区域群、任务组、状态组必须命名，会优先采用功能性命名并受命名规则审查。
- 所有大体量正文资产都优先分布到主线、支线、同伴、区域、Codex、Journal、Items、System，多层共同达成总量，避免把字数偷塞进单一小说正文。
- 若某阶段发现结构冲突，先最小修补上游锚点文件，再继续向下扩写，不带病进入下一层。
