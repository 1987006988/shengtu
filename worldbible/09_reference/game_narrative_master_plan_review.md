# 《升途：封绝之地》大型 CRPG 叙事文本总工程计划 Review

## 1. 文件定位

本文档用于审查 `game_narrative_master_plan.md` 是否足以作为《升途：封绝之地》大型 CRPG 叙事文本总工程的项目级母本。

本 review 只审查总蓝图，不进入批次执行，不生成主线任务文本、同伴文本、支线文本、区域文本、Codex / Journal / Item / System 正文。

## 2. 审查结论总览

- 结论：总蓝图已经足够支撑后续大型 CRPG 叙事文本工程的批次化拆解。
- 范围：仍限定在第一款游戏 `升途：封绝之地` 的 `白鹿原` 阶段，没有无声上浮为整个 `古裂残天` 大全。
- 护栏：保留了白鹿原尺度、白鹿之灵边界、30/31 小说稿降级为上游参考、不得改写宇宙通则与主世界长期正典等关键边界。
- 风险：计划规模极大，不适合作为单轮执行 prompt；必须先拆成批次计划，再分阶段、分包、分资产层执行。
- 后续输入价值：已足以作为 `game_narrative_batch_schedule.md` 的输入母本。

## 3. 检查项

### 3.1 是否已经足够大，不再误收口为短稿 / 单卷

判定：通过。

依据：

- 总蓝图明确最低目标为 `1,500,000` 汉字，目标区间为 `3,000,000–6,000,000` 汉字。
- 计划将文本资产拆成主线、支线、同伴、区域、Codex、Journal、物品、系统反馈、分支状态与失败反馈等多层资产，而不是继续扩写单一小说正文。
- `30_book1_story_draft.md` 与 `31_book1_story_revised.md` 被明确降级为“小说化参考底稿”，不再作为项目终点。

遗留提醒：

- 后续任何验收都不得只看是否有完整故事梗概或章节文本，必须统计各资产层文本量与可调用性。

### 3.2 是否仍限定在白鹿原阶段，不无声上浮成整个 `古裂残天` 大全

判定：通过。

依据：

- 总蓝图明确“以 `白鹿原` 为稳定尺度”。
- 总蓝图明确“只做第一款游戏的白鹿原阶段，不上浮为整个 `古裂残天` 大全”。
- 后续状态机、区域、主支线、同伴与系统反馈均应服务第一款游戏的白鹿原阶段。

遗留提醒：

- 后续批次执行时，如果需要调用 `古裂残天` 长期正典，只能作为上游边界与背景，不得让白鹿原任务结果反向定义整个主世界命运。

### 3.3 是否保留已有护栏和边界

判定：通过。

已保留护栏：

- 不改写宇宙通则。
- 不改写九主世界总层。
- 不改写 `古裂残天` 长期正典。
- 白鹿原是第一款游戏主舞台，不等于整个 `古裂残天`。
- 白鹿之灵是接引法宝之灵的白鹿显化，是主线核心装置的灵性接口。
- 白鹿之灵不得成为万能解释器、万能支援、万能真相机、无代价复生源或世界意志。
- 现有小说稿只作为上游参考，不再作为大型 CRPG 叙事工程终点。

遗留提醒：

- 总蓝图属于项目级执行母本，不等同于新增正典设定；后续写入具体任务、势力状态、区域状态或角色文本时，仍需逐批自检是否触碰正典边界。

### 3.4 是否明确需要再拆成批次执行

判定：通过，但这是当前最重要的执行前提。

依据：

- 总蓝图包含七阶段实施顺序。
- 每阶段包含产出文件、实施标准、最小修补重点与 review 要求。
- 总量目标远超单轮可安全处理范围。

必须执行的下一步：

- 建立 `game_narrative_batch_schedule.md` 或等价批次计划。
- 将阶段 1–7 拆成可验收的小批次。
- 每个批次只处理一个明确资产层或一个明确文件组。
- 每个批次结束后更新 review、状态页、交接页与决策日志。

### 3.5 是否已经足以作为后续 `game_narrative_batch_schedule.md` 的输入母本

判定：通过。

可直接拆批的输入维度：

- 阶段维度：阶段 1 至阶段 7。
- 文件维度：25–29、14–15、32–35、17–20、36–39、40–43、最终验收文件。
- 资产层维度：主线、支线、同伴、区域、Codex、Journal、Item、System。
- 质量维度：范围、正典、白鹿之灵、状态机、文本厚度、任务可实现性、质量、分支、总量。
- 同步维度：`decision_log.md`、`04_current_state.md`、`session_handoff.md`，必要时同步 `glossary.md`。

## 4. 发现的问题与处理建议

### 4.1 问题：计划体量巨大，不能直接执行

说明：总蓝图目标最低 `1,500,000` 汉字，目标区间 `3,000,000–6,000,000` 汉字，并覆盖多个资产层。若直接执行，必然产生失控压缩、重复、遗漏或越界。

处理建议：下一步必须先做批次化重构，生成批次表、批次边界、每批输入文件、输出文件、验收项与同步项。

### 4.2 问题：外部计划中“允许后续进入任务叙事承载层”的句子容易被误解为本轮立即执行

说明：该句应理解为后续阶段授权，不是本轮落库授权。

处理建议：本轮已在总蓝图中明确“本计划落库轮不进入批次执行”。后续必须由批次计划控制进入顺序。

### 4.3 问题：现有仓库已有若干游戏文本工程文件与旧阶段日志，可能与本轮“母本重新落库”产生状态错觉

说明：当前本轮目标是把外部计划正式落库为项目级总蓝图，并为后续批次化执行做准备，不等于宣称既有资产已经达到总蓝图要求。

处理建议：后续批次计划应先做现有资产盘点，区分“已存在的锚点 / 首批底盘”与“尚未达到总量的资产层”。

### 4.4 问题：`09_reference/00_index.md`、`06_stage_gates.md`、`08_pre_task_layer_requirements.md` 存在旧编码显示问题

说明：本轮已回查这些文件以确认范围与阶段口径，但未把编码修复纳入本轮目标。

处理建议：后续如需要长期维护，可单独开一轮“关键护栏文件编码与索引治理”，不要混入本文本生产批次。

## 5. 下一步为什么必须做批次化重构

必须批次化的原因：

- 总量目标过大，单轮无法保证完整、稳定和可审查。
- 资产层不同，主线、同伴、支线、区域与系统反馈需要不同模板和验收法。
- 白鹿原尺度护栏必须在每批次反复检查，不能只在总蓝图中声明一次。
- 状态机、任务文本、Journal、Codex、同伴反馈之间需要逐步互相引用，不能一次性堆砌。
- 每批次都需要同步状态与决策，否则长期协作会失去可追踪性。
